* Purpose

This runbook is the operator handoff for the first Linux Source Producer proof
after the workflow is registered on the default branch.  It does not authorize
merging a pull request, provisioning a runner, changing billing, deleting a
cache, or publishing a release.  Each action requires separate authority.

The producer preserves upstream Gerbil stage ownership.  It invokes only the
stage commands exposed by the exact pinned =build.sh=, preserves their declared
order and dynamic parallelism, and leaves Scheme dependency decisions to
upstream =std/make=.  Checkpoint recovery does not reuse the top-level =make=
entrypoint because that revision's prepare phase deletes the outputs needed for
resume.

Compiler-cache recovery is progressive but non-authoritative.  The workflow
restores the newest compatible cache generation before a source build and
saves a new generation after the build step completes with either =success= or
=failure=.  The primary key rotates by both workflow run ID and run attempt, so
a rerun never tries to overwrite an immutable GitHub cache entry.  A typed
=timed-out= attempt therefore retains its ccache work for an explicitly
authorized retry instead of losing it with the ephemeral runner.

Source-build checkpoint recovery is also progressive and non-authoritative,
but it serves the upstream Scheme frontend rather than the native compiler.
Its restore prefix contains the exact =installDigest= and has no cross-revision
fallback.  A valid generation contains a copy of the exact checkout at one of
the declared completed boundaries =stage1=, =stdlib=, or =tools=.  The live
tree used by the next stage is separate.  A timeout therefore cannot mutate the
safe generation, and only that safe generation is eligible for the rotating
failure-path cache save.

Validation recomputes the generation's canonical file-content manifest and
its SHA-256 receipt binding before use, then recomputes the same manifest after
copying into a sibling temporary source root.  A non-sentinel mutation,
missing file, corrupt manifest, or copy mismatch rejects recovery before the
live source root is replaced.

The top-level upstream =make= is intentionally not used for checkpoint resume:
at this pinned revision its prepare phase deletes =bootstrap/bin|lib= and
=build/bin|lib=.  Resume invokes only the revision-local stage commands exposed
by upstream =build.sh= and preserves their order from output identity.  It does
not implement Scheme dependency or timestamp rules in this repository.

Do not treat that retry as a zero-cache cold-capacity proof.  A partial ccache
generation cannot authorize the Gerbil prefix, and an installation cache entry
is still created only after the bootstrap produces a fresh, exact, validated
=ready= receipt.  A source-build checkpoint also cannot authorize a prefix.
Promotion remains independent of all recovery cache layers.

* Preconditions

Before changing repository or organization settings, verify all of the
following:

- the feature pull request is approved, mergeable, and green;
- the exact expected head SHA is recorded;
- the =Source Producer= workflow is present on =main=;
- the selected runner label is registered and is an explicit Linux x86-64
  source-producer capability;
- the operator has separate authority for runner provisioning and repository
  variable changes.

The known four-core hosted runner does not satisfy the cold 12-minute proof.
Do not dispatch an implicit =ubuntu-latest= cold build as a retry strategy and
do not classify checkpoint-assisted completion as a cold-capacity pass.
The first authorized default-branch dispatch,
[[https://github.com/tao3k/gerbil-bazel/actions/runs/29728030899][Source Producer run 29728030899]] at merge commit
=50f85078393edb74c25569c32dc787054860b5ac=, was explicitly admitted on
=ubuntu-latest= with an installation-cache miss.  Its typed attempt stopped at
exactly 720 seconds in =upstream-build= with =signal=TERM=, no success receipt,
and no materialized capability; the overall job ended after 12 minutes 18
seconds with exit 124.  This is valid guard evidence, not a capacity pass.
After compiler-cache continuity was merged, runs
[[https://github.com/tao3k/gerbil-bazel/actions/runs/29730633648][29730633648]] and
[[https://github.com/tao3k/gerbil-bazel/actions/runs/29731507084][29731507084]]
both restored or advanced ccache yet still spent about 448--449 seconds in
Scheme compilation before the same 720-second =TERM=.  This proves that another
identical ccache-only retry has low evidence value; it is the reason the safe
stage checkpoint exists.

The first hosted stage-checkpoint acceptance used
[[https://github.com/tao3k/gerbil-bazel/actions/runs/29738946094][Source Producer run 29738946094]] at pull-request head
=94e748ebc003c66fbac5f1cd36035a982d6dfc33=.  Attempt 1 stopped at the same
720-second guard and saved the =stage1= boundary in 87,425,458 bytes.  Attempt 2
restored that exact cache, completed =stdlib=, timed out in =libgerbil=, and
advanced the safe boundary to =stdlib= in 124,317,846 bytes.  Attempt 3 restored
=stdlib=, completed the remaining upstream build in 5 minutes 49 seconds, and
closed materialization, activation, producer verification, relocatable
packaging, independent package-consumer validation, capability upload, and
receipt upload in 7 minutes 41 seconds.  It created the authoritative exact
installation cache for
=installDigest=587c0509c35a07228df5b028de6be98841bbfce8c8127f551b43290be4ef5e6b=.
This proves bounded, monotonic checkpoint convergence on the hosted runner; it
does not convert the three-attempt recovery proof into a one-attempt cold
capacity pass and does not authorize promotion.

The internal wrapper sends =TERM= when the 12-minute deadline expires, waits
the declarative 30-second termination grace, and then permits GNU =timeout= to
send =KILL=.  The 30-minute workflow job limit is only the catastrophic outer
ceiling.

* Register the runner selection

Set the repository variable only after the runner label has been provisioned or
verified through the repository or organization runner settings:

#+begin_src sh
repository=tao3k/gerbil-bazel
runner_label='<explicit-linux-x64-runner-label>'
gh variable set GERBIL_SOURCE_RUNNER \
  --repo "$repository" \
  --body "$runner_label"
gh variable list --repo "$repository"
#+end_src

This variable selects scheduling policy.  It is not an attestation of actual
CPU, system, architecture, or memory.  The materialization receipt observes the
actual runner facts after execution.

* Verify registration and dispatch

The workflow must be discoverable from the default branch before dispatch:

#+begin_src sh
repository=tao3k/gerbil-bazel
gh workflow view source-producer.yml --repo "$repository" --ref main
gh workflow run source-producer.yml --repo "$repository" --ref main
#+end_src

Record the run ID and verify that it belongs to the expected =main= head before
watching it:

#+begin_src sh
repository=tao3k/gerbil-bazel
expected_head_sha='<exact-main-head-sha>'
run_id="$(
  gh run list \
    --repo "$repository" \
    --workflow source-producer.yml \
    --branch main \
    --event workflow_dispatch \
    --limit 1 \
    --json databaseId,headSha \
    --jq '.[0].databaseId'
)"
observed_head_sha="$(
  gh run view "$run_id" --repo "$repository" --json headSha --jq .headSha
)"
test "$observed_head_sha" = "$expected_head_sha"
gh run watch "$run_id" --repo "$repository" --exit-status
#+end_src

Do not infer success from the job conclusion alone.  Download all artifacts for
independent receipt verification:

#+begin_src sh
repository=tao3k/gerbil-bazel
audit_dir=".ci/audit/source-producer-$run_id"
mkdir -p "$audit_dir"
gh run download "$run_id" --repo "$repository" --dir "$audit_dir"
#+end_src

* Receipt acceptance

For a genuine first cold-capacity proof, all of the following are mandatory:

- =source-producer-admission.json= has outcome =admitted= and requires a source
  build;
- =gerbil-bootstrap-attempt.json= has outcome =ready=, exit code =0=, no signal,
  =successReceiptPresent=true=, and =successReceiptValidated=true=;
- a current =gerbil-bootstrap-attempt.json= has schema
  =gerbil-bazel.gerbil-bootstrap-attempt.v2= and contains the closed nested
  =checkpoint= snapshot.  A restored attempt records exact restored
  boundary/index/generation; every attempt records the last observed safe
  boundary/index or JSON =false=.  =not-observed= is an observability failure
  that requires investigation even when another field appears healthy;
- the immutable =bootstrap.receipt.json= has outcome =ready= and contains all
  four successful upstream phases;
- the installation materialization is =built=, not =cache-hit=;
- identity, attempt, bootstrap, materialization, manifest, and package evidence
  carry the same =installDigest=;
- the run head SHA is the expected =main= head;
- the capability archive and its digest are present exactly once.

For checkpoint-assisted readiness, additionally verify that the restored
=gerbil-bazel.source-build-checkpoint.v1= receipt carries the same exact
=installDigest=, output-identity digest, source ref, source root, prefix root,
tree-manifest digest, stage sequence, and one declared safe boundary.  The immutable ready receipt
must state the restored boundary and last safe boundary.  The v2 current
attempt receipt must carry restore outcome, restored boundary/index/generation,
and the last observed safe boundary/index derived from progress.  Only
=stage1/4=, =stdlib/5=, and =tools/10= are valid pairs; a restored boundary
cannot be later than the last observed safe boundary.  A checkpoint cache action hit
without a valid typed receipt is a cold fallback, not recovery evidence.

Checkpoint publication and progress publication are separate atomic writes.
Termination between them can leave the attempt snapshot conservatively behind
the authoritative checkpoint pointer, but it cannot make the attempt overclaim
an unobserved boundary.  When they differ, validate the typed checkpoint
receipt and manifest and use that receipt as recovery authority; =pending= and
=not-observed= never authorize materialization or promotion.

A =cache-hit= run is valid materialization and packaging evidence, but it does
not prove that the selected runner can complete a cold source build within 12
minutes.  Do not delete an exact installation cache merely to force a cold
proof without separate destructive-action approval.

A checkpoint-assisted =ready= run is valid built materialization and packaging
evidence after the final immutable receipt closes.  It proves bounded
convergence across exact-identity attempts; it does not prove one-attempt cold
capacity.  Every retry remains an explicit operator action.

The following outcomes prohibit promotion:

| Attempt outcome | Operator decision                                                  |
|-----------------+--------------------------------------------------------------------|
| =failed=        | Inspect =lastPhase= and the preserved exit code.                   |
| =timed-out=     | Treat the runner as insufficient for the 12-minute policy.         |
| =interrupted=   | Resolve cancellation or infrastructure state first.                |
| missing         | Treat the run as unproven; the 30-minute ceiling is not a receipt. |

* Independent promotion

Promotion is a separate manual workflow.  Do not dispatch it until the exact
source run ID, head SHA, =installDigest=, and downloaded evidence have been
independently accepted.

#+begin_src sh
repository=tao3k/gerbil-bazel
run_id='<accepted-source-run-id>'
head_sha='<accepted-source-run-head-sha>'
install_digest='<accepted-install-digest>'
gh workflow run publish-prebuilt.yml \
  --repo "$repository" \
  --ref main \
  -f source_run_id="$run_id" \
  -f head_sha="$head_sha" \
  -f install_digest="$install_digest"
#+end_src

The publish workflow must still independently authorize provenance, validate
the immutable evidence chain, reject an existing release tag, and fail closed
before performing a release mutation.
