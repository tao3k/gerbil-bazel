#+title: RFC 0003: Bazel Artifact Graph and Upstream Gerbil Package Ownership
#+status: Accepted
#+language: en
#+options: toc:2 num:nil

* Revision note

This RFC supersedes the first implementation of a Bazel-owned Gerbil package
identity and revision graph.  That implementation duplicated responsibilities
already owned by Gerbil and was removed before the =gerbil-bazel= v1 API was
stabilized.

The stable boundary is narrower:

- =gerbil.pkg= owns package identity and dependency metadata;
- =gxpkg= owns package installation, update, and lifecycle operations;
- =build.ss= with =std/build-script= and =std/make= owns build membership,
  ordering, and Gerbil-level incrementality;
- Bazel owns declared inputs, isolated staging, toolchain selection, dependency
  tree artifacts, process guards, caching, and receipts.

* Problem

A mutable =~/.gerbil/pkg= directory is not a Bazel dependency graph, but
reimplementing Gerbil's package manager inside Starlark is also incorrect.  It
creates two authorities for package identity, revision selection, dependency
resolution, and build ordering.  The duplicate model inevitably drifts from
=gxpkg= and from project =build.ss= files.

The integration needs atomic Bazel actions and reproducible invalidation
without introducing a second package ecosystem.

* Decision

The public compile API is =gerbil_project_compile=.  A consumer declares the
source closure required by its =build.ss= and any direct Bazel dependency tree
artifacts.  The rule stages those inputs into a writable action root, provides
an isolated =GERBIL_PATH=, and invokes the project's standard build script.

Bazel target names and repository pins identify Bazel artifacts; they are not
Gerbil package identities.  A source archive, =git_override= commit, or local
repository content changes the action key through ordinary declared inputs.
Dependency tree artifacts create the reverse invalidation closure.  No
Starlark rule parses =gerbil.pkg= to select package revisions.

The explicit =@local_gerbil//:install_dependencies= target runs exactly one
standard =gxpkg deps --install= command in the consumer workspace.  It is a
developer-side effect and is never hidden inside a compile action.  Gerbil-
Bazel does not wrap it in a nested =gxpkg env= lifecycle and does not duplicate
package-manager verification with an unconditional package listing.

* Invalidation semantics

| Change | Bazel rebuild set |
|--------+-------------------|
| Source file in project A changes | A and the reverse dependency closure of A |
| Repository pin for dependency A changes | A and the reverse dependency closure of A |
| Unrelated project source changes | The owning project node only |
| Test-only source changes | The affected test action only |
| Native ABI or toolchain input changes | Every action using that toolchain identity |
| Available CPU count changes | The guarded build action whose environment changes |

An identical repeated build executes zero compile actions because Bazel reuses
the complete tree artifact.  Gerbil's own incremental behavior remains useful
inside development and project build workflows, but it is not replaced by a
Bazel-side package scheduler.

* Receipt contract

Every compile action emits one
=gerbil-bazel.project-receipt.v1= JSON value.  The six original required fields
remain stable.  =packageIdentity= and =packageRevision= are reserved v1
compatibility fields and current producers emit empty strings.  They do not
define a package graph or participate in dependency resolution.

Downstream actions consume declared tree artifacts for correctness.  Receipts
provide observability and validation evidence; they are not substitutes for
Bazel dependencies or Gerbil package metadata.

* Toolchain capability closure

The action key includes the registered toolchain and native ABI fingerprint.
The producer compiler is discovered from the selected Gambit installation
unless =GERBIL_CC= is an explicit user override.  Gerbil-Bazel never edits the
installed =gambuild-C= file.

A generated executable-path =GERBIL_GSC= launcher uses Gambit's public =-cc=,
=-cc-options=, and =-ld-options= interfaces.  Repository discovery reads the
literal producer compiler and queries =FLAGS_DYN= and =FLAGS_OBJ= without
writing =gambuild-C=.  Because =-cc= clears the configured mode flags, implicit
or explicit dynamic compilation restores the complete =FLAGS_DYN= value before
adding a platform extra, and =-obj= restores the complete =FLAGS_OBJ= value.
=-c=, =-link=, and =-exe= pass through without a wrapper-added compiler
override.  On Darwin the dynamic extra is
=-Wl,-undefined,dynamic_lookup= and Gerbil's public =GERBIL_GCC= boundary uses
the Xcode Clang driver for the final executable link.  Linux restores the same
producer mode flags without the Darwin extra or a separate final link driver.
The generated =GERBIL_GSC= wrapper is a declared toolchain file and action
input, so a change to normalized producer flags invalidates every consuming
action through ordinary Bazel identity rather than ambient filesystem state.

Both native and prebuilt toolchains derive a positive
=GERBIL_BUILD_CORES= value through the shared host-system resolver.  Resolution
prefers an explicit process override, then a repository environment value, and
otherwise uses the current host's available logical CPU count.  No fixture or
provider hard-codes a machine-specific worker count.

* Acceptance gates

1. A standard fixture must execute =gerbil.pkg -> gxpkg build -> build.ss ->
   std/make=, compile a native module, link its executable, and run it.
2. A repeated identical Bazel build must execute zero compile actions.
3. Changing one declared dependency artifact must invalidate only its reverse
   dependency closure.
4. A failed action must not publish a partial output tree.
5. The Scheme-owned resource guard must terminate a compile action at its
   adaptive deadline and emit guard evidence in the stable receipt.
6. Darwin and Linux toolchain receipts and probes must prove that complete
   producer mode flags are restored, that the Darwin dynamic extra appears
   only on Darwin, and that the wrapper does not take over =-c=, =-link=, or
   =-exe=.

The 2026-07-19 local acceptance passed all 16 =gerbil-bazel= tests.  The focused
standard-package fixture completed in 4.5 seconds, and the seven focused
project/toolchain/dependency gates completed in 9.419 seconds.  POO Flow then
validated real dependency projects through the local library override; the
final receipt consumer completed in 187.355 seconds with dependency actions
served from the Bazel cache.

* Non-goals

- Reimplementing =gxpkg= dependency resolution or tag management.
- Parsing =gerbil.pkg= into a second revision-selection graph.
- Replacing =build.ss=, =std/build-script=, or =std/make= scheduling.
- Introducing a Linux-only compiler graph or GCC-specific optimization policy.
- Mutating the selected Gerbil or Gambit installation.
