#+title: RFC 0004: Stable Gerbil Project Receipt v1
#+startup: overview

* Status

Normative v1.

* Decision

=gerbil-bazel.project-receipt.v1= is the stable public receipt emitted by every
Gerbil project action. Internal execution capabilities do not create a new
project receipt version.

The following six fields are required and retain their original meaning:

- =schema=
- =status=
- =durationSeconds=
- =libraryOutputRequired=
- =packageIdentity=
- =packageRevision=

The two package-named fields are retained to keep the published v1 shape
stable.  Current generic project producers emit empty strings.  They do not
define a second package graph: =gerbil.pkg= and =gxpkg= own package semantics,
while repository pins and Bazel content inputs own reproducible source
identity.

Two optional extension fields are reserved in v1:

- =resourceGuard= contains a typed
  =gerbil-bazel.resource-guard-receipt.v1= value when the action uses the
  Scheme process guard.
- =buildReceipt= contains the build-script-owned JSON value captured through
  =receipt_line_prefix=.

No producer-owned receipt replaces the outer Gerbil-Bazel receipt.

* Compatibility contract

| Concern | v1 rule |
|---------+---------|
| Required identity | The original six fields are always present. |
| Producer JSON | Nested under =buildReceipt=; its schema remains producer-owned. |
| Resource policy evidence | Nested under =resourceGuard=; absent when no guard runs. |
| Unknown top-level fields | Rejected by the normative JSON Schema. |
| New execution capability | Uses a reserved optional field or requires a separately approved project receipt version. |
| Existing reader | A reader consuming only the six required fields remains valid. |

* Atomic cache semantics

The receipt is an output of the same declared project action as the compiled
tree. Declared sources, transitive project tree artifacts, toolchain capability,
and guard implementation participate in the Bazel action key. Repository pins
materialize the revisions declared by the consumer's canonical dependency
manifest. Changing one dependency input invalidates that project and its
reverse-dependent closure; an independent project remains reusable.

* Failure semantics

Failed Bazel actions report their typed guard receipt in the action log because
failed declared outputs are not cacheable. Successful actions store the same
guard value under =resourceGuard=. Host-pressure admission failure, RSS limit,
and deadline termination therefore remain observable without manufacturing a
successful project receipt.

* Upstream-native Darwin compiler and linker capabilities

The stable project receipt version is independent from native toolchain
normalization.  Gerbil-Bazel does not copy or rewrite the installed
=gambuild-C=.  A generated =GERBIL_GSC= launcher uses Gambit's public =-cc= and
=-ld-options= interfaces.  Object generation retains the compiler that produced
the installed runtime; Darwin dynamic modules additionally receive
=-Wl,-undefined,dynamic_lookup=.  Gerbil's public =GERBIL_GCC= boundary selects
Xcode Clang only for the final executable link.  Linux keeps one producer
compiler and receives neither Darwin capability.

This division matters for a Gerbil installation produced by GCC 16: replacing
the producer globally sends GCC-only optimization flags to Clang, while leaving
the final executable link on GCC can fail against Darwin's host link model.
Using the two upstream environment hooks preserves the runtime ABI without
mutating an installed upstream artifact.

The standard regression =//tests/smoke:gxpkg_native_package_test= starts from a
real =gerbil.pkg=, invokes the public =gxpkg= wrapper, delegates to
=std/build-script=, compiles a native =log= reference, links an executable, and
runs it.  It passed in 4.5 seconds.  Seven focused toolchain, runner, receipt,
installer, and gxpkg gates passed in 9.419 seconds.  The real POO Flow
=gerbil-utils= project then passed in 51.606 seconds with a 30.25-second Gerbil
action, and =//scheme:compile_receipt_v1_test= passed in 187.355 seconds with a
181.58-second project critical path.  These receipts were obtained through a
consumer =local_path_override= before any release commit was created.

* Versioning rule

Version 2 is not introduced by this RFC. A future project receipt version needs
a demonstrated incompatible semantic change, a migration plan, parallel reader
coverage, and explicit approval. Field growth alone is not sufficient.
