#+title: RFC 0004: Stable Gerbil Project Receipt v1
#+startup: overview

* Status

Normative v1.

* Decision

=gerbil-bazel.project-receipt.v1= is the stable public receipt emitted by every
Gerbil project action. Internal execution capabilities do not create a new
project receipt version.

The following six fields are required and retain their original meaning:

- =schema=
- =status=
- =durationSeconds=
- =libraryOutputRequired=
- =packageIdentity=
- =packageRevision=

Two optional extension fields are reserved in v1:

- =resourceGuard= contains a typed
  =gerbil-bazel.resource-guard-receipt.v1= value when the action uses the
  Scheme process guard.
- =buildReceipt= contains the build-script-owned JSON value captured through
  =receipt_line_prefix=.

No producer-owned receipt replaces the outer Gerbil-Bazel receipt.

* Compatibility contract

| Concern | v1 rule |
|---------+---------|
| Required identity | The original six fields are always present. |
| Producer JSON | Nested under =buildReceipt=; its schema remains producer-owned. |
| Resource policy evidence | Nested under =resourceGuard=; absent when no guard runs. |
| Unknown top-level fields | Rejected by the normative JSON Schema. |
| New execution capability | Uses a reserved optional field or requires a separately approved project receipt version. |
| Existing reader | A reader consuming only the six required fields remains valid. |

* Atomic cache semantics

The receipt is an output of the same immutable package action as the compiled
tree. Package identity, package revision, declared sources, transitive package
roots, toolchain capability, and guard implementation participate in the Bazel
action key. Changing one dependency revision invalidates that package and its
reverse-dependent closure; an independent package remains reusable.

* Failure semantics

Failed Bazel actions report their typed guard receipt in the action log because
failed declared outputs are not cacheable. Successful actions store the same
guard value under =resourceGuard=. Host-pressure admission failure, RSS limit,
and deadline termination therefore remain observable without manufacturing a
successful project receipt.

* Darwin GCC dynamic-module compatibility

The stable project receipt version is independent from native toolchain
normalization.  A Darwin Gerbil installation built with GCC 16 retains that
producer compiler for ABI-compatible object generation.  Its Gambit
=gambuild-C= overlay augments only =FLAGS_DYN= with
=-Wl,-undefined,dynamic_lookup=.  It does not replace =C_COMPILER=, mutate
=FLAGS_EXE=, or modify the Linux provider.

The distinction is required because Gambit's dynamic-module command compiles C
and links a bundle through the same compiler driver.  Replacing GCC with clang
also passes GCC-only optimization flags to clang.  Applying a global
=LDFLAGS_GAMBUILD= override would affect executable/static-runtime linking.
The dynamic-only overlay preserves both upstream command forms and the installed
runtime ABI while fixing unresolved process-provided symbols in loadable
modules.

#+begin_src mermaid
flowchart LR
  Host["Host system capability"] --> Choice{"Darwin?"}
  Choice -->|"no"| Upstream["Unmodified Gambit compiler and link flags"]
  Choice -->|"yes"| Overlay["Compiler-only gambuild-C overlay"]
  Producer["Installed producer compiler: GCC 16"] --> Overlay
  Overlay --> Obj["Object path: producer compiler, unchanged flags"]
  Overlay --> Dyn["Dynamic path: -bundle plus dynamic_lookup"]
  Overlay --> Exe["Executable/static path: unchanged FLAGS_EXE"]
  Dyn --> ABI["Registered toolchain and Bazel action identity"]
#+end_src

The regression vector =//tests/smoke:native_math_module_test= compiles a real
Gerbil package whose native code references =log=.  Before the overlay, the
Darwin action failed with unresolved =_log= and exit 70 in 33.493 seconds; its
Scheme guard recorded a 318,717,952-byte process-tree peak.  After the
dynamic-only overlay, the native package, stable v1 receipt, and toolchain
environment gates passed.  The complete Bazel suite passed 15 of 15 tests in
110.172 seconds.  The prebuilt and Darwin auto repository-provider receipts
also passed, in 72 and 88 seconds respectively.  These are compatibility-gate
times, not new project cold-build baselines.

* Versioning rule

Version 2 is not introduced by this RFC. A future project receipt version needs
a demonstrated incompatible semantic change, a migration plan, parallel reader
coverage, and explicit approval. Field growth alone is not sufficient.
