#+title: RFC 0007: Source Build Identity v1
#+author: gerbil-bazel maintainers
#+language: en
#+options: toc:2 num:t

* Status

Proposed.

This RFC defines the cache identity for an upstream Gerbil source installation
and its compiler-cache namespace. The schema and configuration contract remain
at v1 until the complete identity is stable.

* Evidence

The Linux installation cache currently contains a successful build receipt for
Gerbil revision =07c8481588a8b07dbf05832687817cd398902ac0=. That build used four
available logical CPUs, observed 680 ccache compiler operations, and took 1,381
seconds. The producing CI step ran from 19:54:40Z to 20:17:41Z on 2026-07-18.

The previous installation cache key included the digest of the complete
=bootstrap_gerbil.sh=. Therefore a receipt-only or logging-only script change
invalidated a native installation even when the source, compiler, libraries,
and output-affecting configure arguments were unchanged.

* Identity split

The compiler-cache namespace excludes the Gerbil source revision because
ccache validates compiler inputs by content. A revision change may therefore
reuse valid object entries while producing a distinct installation digest.
The namespace includes the ccache path and version so incompatible cache
storage implementations cannot share a namespace; those fields do not affect
the installation digest.

The installation digest includes:

- exact source revision and source URL;
- operating system and architecture;
- compiler command, resolved path, and version;
- linker path and version;
- OpenSSL, SQLite, and zlib package versions;
- output-affecting build environment;
- the digest of the declarative =outputIdentity= configuration.

The execution policy includes available-host parallelism, ccache capacity, and the 12-minute source-build timeout.
Changing execution policy or observability MUST change the full configuration
digest but MUST NOT invalidate an otherwise identical native installation.  The
timeout therefore does not contribute to =installDigest=.

CI selects the requested source-producer runner from the
=GERBIL_SOURCE_RUNNER= repository variable and defaults to =ubuntu-latest=.
This scheduling label is execution policy, not output identity; the current-run
materialization receipt observes the actual runner facts separately.

GitHub MUST register the =Source Producer= workflow from the default branch
before an explicit dispatch can target another ref.  A cache scoped to a
pull-request merge ref is not evidence that a branch or default-branch
dispatch can restore it.  The post-restore admission gate uses only the exact
cache result and the declared =GERBIL_SOURCE_RUNNER= selection: cache hits are
admitted, cold misses require an explicit registered runner label, and an
implicit default-runner cold miss is blocked before source compilation.
Admission does not duplicate or predict runner capacity facts.

* Upstream ownership

Gerbil upstream continues to own configure, build scheduling, bootstrap stages,
and installation. The identity layer does not invoke private stage scripts and
does not reproduce gxpkg or =std/make= semantics.

* Atomic invalidation scenarios

| Scenario                                  | Installation digest | Compiler-cache namespace |
|-------------------------------------------+---------------------+--------------------------|
| Same inputs                               | unchanged           | unchanged                |
| Gerbil revision changes                   | changes             | unchanged                |
| Output configure arguments change         | changes             | changes                  |
| Compiler or native library changes        | changes             | changes                  |
| ccache implementation changes             | unchanged           | changes                  |
| ccache capacity changes                   | unchanged           | unchanged                |
| Receipt or logging implementation changes | unchanged           | unchanged                |

The checked-in identity test MUST prove every row without compiling Gerbil.
The next real CI cache miss is then a controlled “empty installation plus warm
compiler cache” scenario, not an unclassified cold retry.

* Receipts

The source-build identity receipt is producer evidence.  A restored bootstrap
receipt remains immutable evidence of the build that produced the installation;
it must not be rewritten to claim that the current run observed a cache hit.

Before materialization, the producer emits
=gerbil-bazel.source-producer-admission.v1=.  It records the exact cache result,
whether the runner label was explicit, whether a source build is required, and
the admitted or blocked decision.  It MUST NOT record inferred CPU, system, or
architecture facts; those remain owned by the materialization receipt.  An
explicit runner admission remains subject to the identity-owned 12-minute
source-build guard.

The current run emits =gerbil-bazel.install-materialization-receipt.v1=.  Its
=materialization= field is exactly =cache-hit= or =built=, and its
=installDigest= must match the embedded source-build identity in the producer
bootstrap receipt.  The receipt also records the current runner name, normalized
system and architecture, and available logical CPU count.  These facts describe
materialization; they do not rewrite producer evidence or contribute to
=installDigest=.  This separates production from materialization without
duplicating upstream build semantics.

Packaging copies the source-build identity receipt beside the archive.  The
capability manifest and package receipt both carry the same =installDigest=.
The package receipt also records the identity receipt SHA-256, while the archive
SHA-256 continues to prove byte integrity.  Promotion succeeds only when the
identity receipt, manifest, package receipt, archive digest, and requested
release identity form one closed chain.

Promotion also emits =gerbil-bazel.prebuilt-promotion-authorization.v1=.  An
explicit =Source Producer= dispatch can authorize a successful =main= producer
or a non-=main= run bound to exactly one open same-repository pull request with
base =main= and an exact head SHA.  The authorization derives the branch from
the successful producer run and queries GitHub for the unique pull request; it
does not trust a caller-supplied pull-request number.  The authorization
receipt is validated with the capability evidence and attached to the
immutable release; a missing, forked, closed, wrong-base, ambiguous, or
SHA-drifted producer provenance fails closed.

Consumers declare both the expected =install_digest= and archive =sha256=.
The repository rule fails closed when the manifest identity differs; it never
falls back to a source build.  The =auto= tag uses
=linux_prebuilt_install_digest= for the same Linux capability contract.

=gerbil-bazel.source-build-identity.v1= records the complete inputs and both
digests. The bootstrap receipt MUST record the identity and phase durations for
source preparation, configure, upstream build, and install. Generated receipts
remain CI artifacts under =.ci= and are never committed.
