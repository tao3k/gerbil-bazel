#+title: RFC 0002: Immutable Gerbil Toolchain Capabilities
#+author: gerbil-bazel maintainers
#+language: en

* Status

- Decision: Accepted
- Scope: Darwin host capabilities and immutable Linux prebuilt capabilities
- Package graph: RFC 0006

* Decision

gerbil-bazel separates Gerbil toolchain acquisition from Gerbil package graph
evaluation.

- =gerbil.host= resolves an explicitly installed native capability.
- =gerbil.prebuilt= imports an immutable capability archive.
- =gerbil.auto= selects host mode on Darwin and prebuilt mode on Linux.

All three modes register the same =GerbilToolchainInfo= contract. They do not
declare a package root, package library view, source graph, or build program.

* Capability contract

A registered capability exposes:

- =gxi=, =gxc=, =gxpkg=, and =gxtest= executables;
- compiler, assembler, and linker paths;
- normalized process environment;
- native ABI fingerprint;
- Gerbil and Gambit producer options;
- immutable dependency library roots owned by the capability itself;
- available logical CPU count and physical memory;
- a machine-readable toolchain receipt.

Host CPU count, memory, SDK paths, compiler paths, and link flags are discovered
per machine. They are never fixed to one developer workstation or CI runner.

* Darwin host mode

Darwin resolves the coherent Xcode developer directory and SDK through the host
adapter. Homebrew formulae may contribute include, library, linker, and
=pkg-config= paths.

The native installation remains immutable. gerbil-bazel wraps the public Gambit
compiler interface instead of rewriting =gambuild-C=. Dynamic, object, and final
link modes preserve the producer flags while selecting the current Xcode Clang
toolchain.

The supported distribution path is =tao3k/gerbil/gerbil-scheme= at the pinned
Gerbil revision used by this repository.

* Linux prebuilt mode

The prebuilt declaration requires:

- one installation digest;
- one archive SHA-256;
- one or more immutable URLs;
- the declared archive architecture;
- a capability manifest inside the archive.

Repository evaluation verifies the archive digest, installation digest,
platform, architecture, tool paths, capability manifest schema, and executable
runtime probes before registering the toolchain.

Regular Linux consumers do not compile Gerbil. Capability production is a
separate release workflow.

* Capability production

Linux capability production follows the upstream Gerbil build:

1. =./configure=;
2. parallel =make= using the discovered available CPU count;
3. =make install=;
4. relocation and runtime verification;
5. deterministic archive creation;
6. capability manifest and digest emission.

A compiler cache may wrap the standard =CC= interface. The producer does not
replace the upstream Make graph or impose a version-specific GCC preference.

Promotion is explicit and fail-closed. A release must be authorized by a
successful producer run for the exact target commit and platform. The immutable
archive, installation digest, archive SHA-256, and authorization receipt are
published together.

* Public Bzlmod API

#+begin_src starlark
gerbil = use_extension("@gerbil_bazel//gerbil:extensions.bzl", "gerbil")

gerbil.auto(
    name = "local_gerbil",
    expected_version_prefixes = ["Gerbil v0.18.2"],
    linux_prebuilt_arch = "x86_64",
    linux_prebuilt_install_digest = "<installation digest>",
    linux_prebuilt_sha256 = "<archive sha256>",
    linux_prebuilt_urls = ["<immutable release URL>"],
)

use_repo(gerbil, "local_gerbil")
register_toolchains("@local_gerbil//:registered_toolchain")
#+end_src

Explicit =gerbil.host= and =gerbil.prebuilt= declarations expose the same
capability-specific fields without automatic provider selection.

* Bzlmod lock

The extension is operating-system and architecture dependent. The checked-in
=MODULE.bazel.lock= must retain the Linux and Darwin evaluations required by CI.

Lock refresh is a controlled cross-platform workflow:

1. seed the Linux evaluation;
2. merge the Darwin evaluation;
3. verify the merged lock again on Linux.

Ordinary CI uses =--lockfile_mode=error= and never mutates the lock.

* Package graph boundary

Toolchain repositories do not inspect =gerbil.pkg= and do not materialize
package dependency state. RFC 0006 owns the canonical package graph, immutable
dependency locks, generated package actions, and package receipts.

This separation prevents a capability provider from depending on the graph that
needs the capability for repository-phase manifest evaluation.

* Required validation

- host and prebuilt runtime version probes;
- native ABI and producer-option receipts;
- Darwin compiler/linker mode probes;
- Linux archive and installation digest mismatch rejection;
- architecture mismatch rejection;
- immutable tool wrapper behavior;
- adaptive CPU and memory discovery;
- generated dependency-installer compile wiring;
- cross-platform Bzlmod lock verification;
- package graph bootstrap proof through RFC 0006.
