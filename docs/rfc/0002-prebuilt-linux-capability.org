#+title: RFC 0002: Immutable Prebuilt Linux Gerbil Capability
#+author: gerbil-bazel maintainers
#+language: en

* Nested tool discovery

The provider environment prepends the selected capability's tool directory to
=PATH= before the first raw =gxpkg= process starts.  It also derives
=GERBIL_GSC= from the relocated =GERBIL_HOME=, appends Gambit =GAMBOPT=
runtime mappings for =~~=, =~~bin=, and =~~lib=.  The registered toolchain
leaves the installed =gambuild-C= immutable.  Repository discovery reads its
literal =C_COMPILER= declaration and queries =FLAGS_DYN= and =FLAGS_OBJ=
without writing the installation.  A repository-owned executable-path
=GERBIL_GSC= launcher uses Gambit's public =-cc=, =-cc-options=, and
=-ld-options= interfaces.  Because =-cc= clears the configured mode flags,
implicit or explicit dynamic compilation restores the complete =FLAGS_DYN=
value and =-obj= restores the complete =FLAGS_OBJ= value.  Darwin appends its
dynamic-link extra after =FLAGS_DYN= and selects Xcode Clang through
=GERBIL_GCC= for the final executable link.  Linux restores the same producer
mode flags without the Darwin extra or a separate final link driver.
Dependency build scripts launched through
=/usr/bin/env gxi= and compiler processes launched by =gxc= therefore resolve
the same immutable prebuilt capability instead of retaining producer-machine
paths.  The synthetic prebuilt fixture requires the nested =gxi= lookup to
succeed, while capability production requires a relocated =gxc -> gsc=
compilation to succeed.

* Status

- Decision: Accepted
- Implementation: Prebuilt provider published; auto provider validated on Linux and Darwin
- Scope: =gerbil-bazel= library and its Linux release pipeline
- Consumer: Bzlmod projects, including =poo-flow=

* Context

=gerbil-bazel= already owns Gerbil toolchain discovery, registration, and rule
consumption. A consumer registers a toolchain repository through the module
extension, while project rules resolve =GerbilToolchainInfo= and keep
=build.ss= as the semantic build owner.

The native =gerbil.host(...)= tag is appropriate for local development and for
macOS installations supplied by Homebrew. It is not a Linux cold-start
solution: a fresh Linux runner still has to compile Gerbil before Bazel can
register the native installation. A GitHub Actions cache shortens repeated runs
inside one repository, but it cannot act as a stable cross-repository
capability for downstream consumers.

* Decision

Capability promotion MUST be an explicit, receipt-producing operation.  It MAY
authorize either a successful =main= producer run or a successful run for an
open same-repository pull request whose base is =main= and whose head SHA is the
exact requested release target.  Fork runs, non-main bases, closed pull
requests, SHA drift, and duplicate or expired artifacts MUST fail closed.

The authorization decision emits
=gerbil-bazel.prebuilt-promotion-authorization.v1=.  This receipt is attached to
the immutable release beside the archive evidence, so publication authority is
auditable without weakening the independent =installDigest= and archive
SHA-256 proofs.

The library SHALL expose a separate =gerbil.prebuilt(...)= tag. It imports an
immutable, relocatable Gerbil archive by URL and SHA-256, validates the embedded
capability manifest, matches the archive to the execution platform, and
registers the same public Gerbil toolchain contract used by the native host
path.

The ordinary cross-platform consumer API SHALL be =gerbil.auto(...)=. The
module extension reads =module_ctx.os= and selects exactly one repository
provider before repository creation: native =host= on Darwin and immutable
=prebuilt= on Linux. It SHALL reject unknown systems and a declared Linux
archive architecture that does not match the consumer host. It SHALL NOT
instantiate both providers and defer selection to toolchain resolution.

Regular CI SHALL be a capability consumer only.  The repository root uses
=gerbil.auto(...)= with the Homebrew host capability on Darwin and the
digest-addressed immutable release on Linux.  Regular CI SHALL NOT bootstrap
Gerbil source, materialize a producer prefix, package an archive, or upload a
producer artifact.  The pinned Linux capability has installation digest
=ee5a236818e1b8fd9cc6e886b510795495e95402e790892961e6a1059a5c3208= and
archive SHA-256
=741d67f1d1a37e057181bfe8b3348ce2a9cfce06ac3b9c4e914400978f9d5fe7=.

Because provider selection observes both =module_ctx.os.name= and
=module_ctx.os.arch=, the public module extension SHALL be declared
=os_dependent=True= and =arch_dependent=True=. A checked-in Bzlmod lockfile
must retain operating-system- and architecture-specific extension evaluations;
it must never replay a Darwin host repository specification on Linux, a Linux
prebuilt repository specification on Darwin, or a Linux prebuilt repository
specification across incompatible architectures.

Homebrew remains the macOS distribution path. Linux capability production and
consumption belong to =gerbil-bazel=. =poo-flow= SHALL consume the registered
library capability and SHALL NOT implement another source bootstrap, direct
=gxc= path, or Homebrew-based Linux adapter.

Linux capability production SHALL retain the pinned upstream Gerbil build:
=./configure=, parallel =make= using the discovered available CPU count, and
=make install=.  A compiler cache may wrap the system compiler through the
standard =CC= contract, but the producer SHALL NOT replace the upstream build
graph or impose a versioned GCC preference.  Consumers honor an explicit
=GERBIL_CC= override and otherwise select the platform =cc= capability.

Darwin host consumption remains a distinct platform adapter.  It discovers a
coherent =DEVELOPER_DIR= and =SDKROOT=, resolves the linker through =xcrun=,
and projects Homebrew include, library, and pkg-config paths.  These Darwin
requirements SHALL NOT be copied into the Linux source-build path.

* Public Bzlmod API

Every prebuilt Linux declaration carries two independent proofs:
=install_digest= identifies the exact source/configuration/native build inputs,
while =sha256= identifies the downloaded archive bytes.  The =auto= form uses
=linux_prebuilt_install_digest= for the same contract.  A mismatch fails closed
before toolchain registration and never invokes a source-build fallback.

Cross-platform consumers declare one capability repository:

#+begin_src starlark
gerbil = use_extension("@gerbil_bazel//gerbil:extensions.bzl", "gerbil")
gerbil.auto(
    name = "local_gerbil",
    expected_version_prefixes = ["Gerbil 07c8481", "Gerbil v0.18.2"],
    linux_prebuilt_arch = "x86_64",
    linux_prebuilt_install_digest = "<64 hexadecimal characters>",
    linux_prebuilt_sha256 = "<64 hexadecimal characters>",
    linux_prebuilt_urls = [
        "https://github.com/tao3k/gerbil-bazel/releases/download/<digest-addressed-capability>/<archive>",
    ],
    project_dependency_packages = ["clan", "gslph"],
    project_root_marker = "//:MODULE.bazel",
)
use_repo(gerbil, "local_gerbil")
register_toolchains("@local_gerbil//:registered_toolchain")
#+end_src

The explicit Linux-only provider remains available:

#+begin_src starlark
gerbil = use_extension("@gerbil_bazel//gerbil:extensions.bzl", "gerbil")
gerbil.prebuilt(
    install_digest = "<64 hexadecimal characters>",
    name = "gerbil_linux_x86_64",
    expected_version_prefixes = ["0.18.2"],
    sha256 = "<64 hexadecimal characters>",
    urls = [
        "https://github.com/tao3k/gerbil-bazel/releases/download/<capability>/<archive>",
    ],
)
use_repo(gerbil, "gerbil_linux_x86_64")
register_toolchains("@gerbil_linux_x86_64//:registered_toolchain")
#+end_src

The =auto=, =host=, and =prebuilt= tags are distinct capability declarations.
Repository names are unique across all tag classes. The =auto= tag instantiates
one of the existing repository providers; it does not add another toolchain
contract. The prebuilt path never falls back to host discovery or a source
build when download, digest, manifest, platform, or runtime validation fails.

* Project-library view contract

All three providers accept the same optional project-library declaration:
=project_root_marker= identifies the consumer project, the relative library
root defaults to =.gerbil/lib=, and =project_dependency_packages= names the
packages projected into the repository as =lib/<package>=. The repository
receipt records =dependencyPolicy=project-library-view= and a
=dependencyState= map containing =ready= or =missing= for each declaration.

The Linux prebuilt provider therefore imports immutable Gerbil runtime bytes
while exposing consumer-owned package sources through Bazel repository inputs.
It does not bake downstream packages into the archive and it does not perform
an implicit =gxpkg= installation. Dependency installation remains an explicit,
separate capability.

Every provider repository publishes =install_dependencies=. Its launcher uses
the selected provider environment, enters the consumer workspace, and runs the
standard =gxpkg deps --install= command exactly once. When installation changes
a watched project package from absent to present, the next Bazel command
re-evaluates the repository and the receipt changes that package from =missing=
to =ready=. This lifecycle is validated by an isolated synthetic prebuilt
consumer; consumers do not need provider-specific install scripts.  The target
is an explicit developer action, not an input-producing step hidden inside a
compile rule.

The provider environment SHALL be applied by the host =env= executable to the
single raw =gxpkg= process.  It SHALL NOT create a nested =gxpkg env env=
lifecycle or perform an unconditional package listing.  The selected
=GERBIL_HOME= is already visible when =gxpkg= imports =:gerbil/core=.

* Archive and manifest contract

The archive root contains =gerbil-bazel-capability.json= and a relocatable
=prefix/= tree. The archive SHA-256 is the capability identity supplied to
Bazel; therefore it covers both the manifest and every runtime byte.

  "nativeAbiFingerprint": "<40 hexadecimal characters>",

All manifest paths are relative and traversal-free. The repository rule rejects
unknown schemas, unsafe or missing paths, a non-hexadecimal digest, a platform
mismatch, a runtime version mismatch, and an ABI fingerprint of the wrong
shape. Available CPU count, physical memory, compiler, assembler, and linker
remain consumer-host capabilities and are discovered adaptively rather than
frozen into the producer artifact.

* Producer lifecycle

The promotion workflow is manually dispatched with an exact source run ID,
head SHA, and =installDigest=.  It fetches the run, pull-request metadata when
applicable, and the artifact inventory once; then it validates the complete
authorization frontier before downloading or publishing capability bytes.

The existing pinned, upstream-parallel Linux bootstrap remains the rare
producer. After the source build and public Bazel API validation succeed, the
producer:

The source producer is a separate =workflow_dispatch=-only workflow.  A
non-=main= dispatch is promotable only when its runtime branch resolves to
exactly one open same-repository pull request with base =main= and the run head
SHA equals the pull-request head SHA.  It fails closed for a missing,
ambiguous, forked, closed, wrong-base, or SHA-drifted pull request.  This keeps
the producer available for deliberate capability refreshes without placing a
cold source build on the regular consumer-CI path.

The =Source Producer= workflow MUST first be registered from the default
branch before GitHub can accept an explicit dispatch.  A cache created under a
pull-request merge ref MUST NOT be treated as dispatch-accessible branch or
default-branch state.  After the exact installation-cache lookup, the producer
emits =gerbil-bazel.source-producer-admission.v1=.  A complete cache hit is
admitted on the selected runner.  A cold miss is admitted only when
=GERBIL_SOURCE_RUNNER= explicitly selects a registered runner label; an
implicit =ubuntu-latest= cold miss is blocked immediately with the admission
receipt.  Admission is not a capacity attestation: the existing 12-minute
build guard and the later materialization receipt remain the execution proof.

2. rejects absolute symlinks;

Pull-request CI uploads these files only as verification evidence. A release
workflow may publish them only after the relocation and synthetic consumer
receipts pass. Downstream modules consume a release asset with a fixed digest,
never an expiring workflow artifact.

* Performance and failure gates

- A downstream Linux consumer MUST NOT compile Gerbil.
- A missing or invalid prebuilt capability MUST fail closed; source fallback is
  forbidden in the consumer lane.
- Fresh-runner download, registration, and smoke verification SHOULD complete
  within 120 seconds.
- Repository-cache hits SHOULD reduce registration to metadata validation and
  normal Bazel analysis time.
- Source production remains guarded separately and is not counted as consumer
  cold-start time.
- Receipts record archive identity, platform, version, adaptive host resources,
  runtime verification, and elapsed time.

* Compatibility and migration

The existing =gerbil.host(...)= and =gerbil.prebuilt(...)= APIs remain
compatible, with the project-library declaration added uniformly to every
provider. Projects may use them for explicitly single-provider environments. A
cross-platform project migrates to one =gerbil.auto(...)= registration;
compile, test, and development targets remain unchanged because every path
resolves the same toolchain provider.

The migration is complete when a published Linux archive passes relocation,
the =gerbil.prebuilt(...)= consumer test, and =poo-flow= compile/test receipts
without a Gerbil source-build phase.
